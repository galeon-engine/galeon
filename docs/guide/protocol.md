# Protocol — Boundary Abstraction

Galeon's protocol layer lets the same game logic work in-process, over HTTP/WS,
or through native bindings. Define your types once in Rust; Galeon generates
TypeScript types, route descriptors, and adapter glue.

## Protocol Items

| Kind | Purpose | Macro |
|------|---------|-------|
| Command | State-changing request | `#[galeon::command]` |
| Query | Read-only request | `#[galeon::query]` |
| Event | Fact emitted after state change | `#[galeon::event]` |
| DTO | Boundary data structure | `#[galeon::dto]` |

```rust
#[galeon_engine::command]
pub struct SpawnUnit {
    pub unit_id: u64,
    pub location_id: u64,
}

#[galeon_engine::query]
pub struct GetWorldSnapshot;

#[galeon_engine::event]
pub struct UnitDestroyed {
    pub unit_id: u64,
    pub destroyed_at: String,
}

#[galeon_engine::dto]
pub struct WorldSnapshot {
    pub units_active: u32,
    pub units_idle: u32,
}

#[galeon_engine::command(surface = "authority")]
pub struct AdminReset {
    pub zone_id: u64,
}

#[galeon_engine::dto(surfaces = ["authority", "gameplay"])]
pub struct UnitView {
    pub unit_id: u64,
    pub name: String,
}
```

## Manifest

`ProtocolManifest::collect("my-protocol@0.1")` gathers all registered items
into a machine-readable JSON manifest. This is the input to code generation.

Unannotated items land in the manifest's `default_surface`, which defaults to
`"default"`. Multi-surface projects can rename that implicit surface with
`ProtocolManifest::collect_with_default_surface("my-protocol@0.1", "gameplay")`.

Surface rules:

- `surface = "authority"` assigns an item to one named surface
- `surfaces = ["authority", "gameplay"]` shares an item across named surfaces
- no surface annotation means "default surface only"

That keeps single-surface projects zero-config while making shared DTOs, events,
queries, and commands explicit instead of globally flattened.

## TypeScript Generation

`generate_typescript(&manifest)` emits TypeScript interfaces:

```typescript
// Auto-generated by Galeon Engine — do not edit.
export interface SpawnUnit {
  unit_id: number;
  location_id: number;
}

export type GetWorldSnapshot = Record<string, never>;

export interface WorldSnapshot {
  units_active: number;
  units_idle: number;
}
```

Type mapping: `u64`/`f32` → `number`, `String` → `string`, `bool` → `boolean`,
`Vec<T>` → `T[]`, `Option<T>` → `T | null`, `HashMap<K,V>` → `Record<K,V>`.

## Protocol Descriptors

`generate_descriptors(&manifest)` produces route metadata grouped by surface:

| Kind | Default route | HTTP method |
|------|--------------|-------------|
| Command | `/commands/<kebab-name>` | POST |
| Query (no fields) | `/queries/<kebab-name>` | GET |
| Query (with fields) | `/queries/<kebab-name>` | POST |
| Event | `/events/<kebab-name>` | GET (stream) |

```rust
let manifest = ProtocolManifest::collect_with_default_surface(
    "my-protocol@0.1",
    "gameplay",
);
let descriptors = generate_descriptors(&manifest);

for surface in &descriptors.surfaces {
    println!("surface: {}", surface.name);
    for descriptor in &surface.descriptors {
        println!("  {} {}", descriptor.name, descriptor.route);
    }
}
```

Routes remain relative to a surface. Adapters can mount each surface under their
own base path without flattening unrelated commands, queries, or events into one
project-wide descriptor list.

## Handler Registry

`HandlerRegistry` is the seam between generated adapters and game logic:

```rust
use galeon_engine::handler::{CommandHandler, QueryHandler, HandlerRegistry};

struct MyHandler;

impl CommandHandler<SpawnUnit, SpawnResult> for MyHandler {
    fn handle(&self, cmd: SpawnUnit) -> Result<SpawnResult, String> {
        // game logic here
        Ok(SpawnResult { ok: true })
    }
}

let mut registry = HandlerRegistry::new();
registry.register_command::<SpawnUnit, SpawnResult, _>(MyHandler);

// Local adapter (in-process, typed)
let result: SpawnResult = registry.dispatch_command(cmd).unwrap();

// Remote adapter (JSON boundary, keyed by stable protocol name)
let json = registry.dispatch_command_json("SpawnUnit", request_json).unwrap();
```

Both adapters target the same registry — same handlers, two execution modes.
